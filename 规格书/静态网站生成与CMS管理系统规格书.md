## 项目概述

### 1.1 项目背景

本项目旨在建立一套完整的静态网站生成、管理与部署系统，通过AI辅助生成、CMS集中管理、Git版本控制以及云端部署，实现高效的多站点管理流程。

### 1.2 系统架构

```
┌─────────────────┐
│  AI生成工具     │
│  (Cursor)       │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  静态网页文件   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐      ┌──────────────┐
│  GitHub仓库     │◄─────┤  CMS后台     │
│  (版本控制)     │      │  (Strapi)    │
└────────┬────────┘      └──────────────┘
         │
         ▼
┌─────────────────┐
│  云端托管平台   │
│ (Vercel/Netlify)│
└─────────────────┘
```

### 1.3 核心目标

- **高效生成**：通过AI辅助快速生成符合规范的静态网页
- **集中管理**：通过CMS系统统一管理多站点内容
- **版本控制**：使用GitHub进行代码版本管理
- **快速部署**：自动化部署流程，提升发布效率

## 静态网页生成方式

本系统支持三种静态网页生成方式，可根据项目需求选择合适的方法。

### 2.1 方式一：AI直接生成

#### 2.1.1 流程说明

**适用场景：** 全新网站、无特定模板要求

**操作流程：**
1. 在Cursor中直接向AI提供需求描述
2. 明确说明网站主题、风格、功能需求
3. AI根据提示词生成完整的静态网页代码
4. 检查生成结果是否符合要求

**优点：**
- 生成速度快
- 灵活性高
- 适合快速原型开发

**注意事项：**
- 需要提供清晰、详细的需求描述
- 生成后需进行质量检查
- 可能需要多次迭代优化

#### 2.1.2 提示词要求

- 明确网站主题和定位
- 说明所需页面结构（首页、关于、文章列表等）
- 指定设计风格（现代、简约、商务等）
- 列出必需功能模块

---

### 2.2 方式二：基于HTML模板生成

#### 2.2.1 流程说明

**适用场景：** 需要保持特定架构结构、有现成模板

**操作流程：**
1. 下载或准备HTML静态网页模板
2. 将模板文件提供给Cursor
3. 向AI说明网站主题
4. 提供文章内容要求：
   - 文章必须包含的重点内容
   - 文章格式要求（标题层级、段落结构等）
   - 内容长度要求
5. AI根据模板结构生成符合架构的静态网页
6. 验证生成结果与模板结构的一致性

**优点：**
- 保持统一的网站架构
- 结构规范，易于维护
- 适合批量生成相似结构的网站

**注意事项：**
- 模板质量直接影响生成效果
- 需要确保模板结构清晰
- 生成后需验证结构一致性

#### 2.2.2 模板要求

- 模板应包含完整的HTML结构
- 建议包含CSS样式文件
- 模板应具备良好的代码注释
- 模板应遵循响应式设计原则

---

### 2.3 方式三：基于Figma设计稿生成

#### 2.3.1 流程说明

**适用场景：** 有完整设计稿、需要精确还原设计

**操作流程：**
1. 在Figma中完成网站设计
2. 将Figma设计稿导出为图片（PNG/JPG）
3. 将设计图片提供给Cursor
4. 向AI说明需要根据设计稿生成静态网页
5. 提出需要新增的功能项目（如：每日文章更新模块）
6. AI根据设计稿样式生成对应的静态网页
7. 对比生成结果与设计稿的一致性

**优点：**
- 设计还原度高
- 视觉呈现精确
- 适合有明确设计要求的项目

**注意事项：**
- 设计稿应清晰完整
- 需要提供设计规范说明
- 生成后需进行视觉对比检查

#### 2.3.2 设计稿要求

- 设计稿应包含所有页面状态
- 建议提供设计规范（颜色、字体、间距等）
- 标注交互说明
- 提供响应式断点设计

|---------|---------|---------|-----------|-----------|

## 静态网页质量检查

生成静态网页后，需要进行全面的质量检查，确保网站符合发布标准。

### 3.1 检查项目清单

#### 3.1.1 结构检查

**HTML结构验证：**
- [ ] DOCTYPE声明正确
- [ ] HTML标签闭合完整
- [ ] 语义化标签使用正确（header, nav, main, article, footer等）
- [ ] 页面层级结构清晰
- [ ] 链接路径正确（相对路径/绝对路径）
- [ ] 图片路径正确
- [ ] 脚本文件引用正确

**文件结构验证：**
- [ ] 文件命名规范统一
- [ ] 目录结构清晰合理
- [ ] CSS文件组织规范
- [ ] JavaScript文件组织规范
- [ ] 资源文件分类存放

#### 3.1.2 响应式设计检查

**断点测试：**
- [ ] 桌面端显示正常（1920px, 1440px, 1280px）
- [ ] 平板端显示正常（768px, 1024px）
- [ ] 移动端显示正常（375px, 414px）
- [ ] 横屏/竖屏切换正常

**布局检查：**
- [ ] 内容不溢出容器
- [ ] 文字大小适配不同屏幕
- [ ] 图片自适应显示
- [ ] 导航菜单在小屏幕正常显示
- [ ] 表格在小屏幕可横向滚动

**交互检查：**
- [ ] 按钮点击区域足够大（移动端）
- [ ] 表单输入框大小合适
- [ ] 触摸操作流畅

#### 3.1.3 内容检查

**文章内容：**
- [ ] 文章标题清晰明确
- [ ] 文章内容完整，包含所有必需重点
- [ ] 段落结构合理
- [ ] 标题层级正确（H1-H6）
- [ ] 列表格式正确
- [ ] 引用格式正确
- [ ] 图片alt属性完整

**页面内容：**
- [ ] 首页内容完整
- [ ] 关于页面信息准确
- [ ] 联系页面表单功能正常
- [ ] 文章列表显示正确
- [ ] 文章详情页内容完整

#### 3.1.4 性能检查

**加载速度：**
- [ ] 首屏加载时间 < 3秒
- [ ] 图片优化（WebP格式、适当压缩）
- [ ] CSS/JS文件压缩
- [ ] 无冗余代码

**资源优化：**
- [ ] 图片懒加载
- [ ] 字体文件优化
- [ ] 第三方资源CDN加载

#### 3.1.5 兼容性检查

**浏览器兼容：**
- [ ] Chrome最新版
- [ ] Firefox最新版
- [ ] Safari最新版
- [ ] Edge最新版
- [ ] 移动端浏览器（iOS Safari, Chrome Mobile）

**功能兼容：**
- [ ] JavaScript功能正常
- [ ] CSS动画效果正常
- [ ] 表单提交功能正常

#### 3.1.6 SEO检查

**基础SEO：**
- [ ] 页面title标签完整
- [ ] meta description设置
- [ ] meta keywords设置（如需要）
- [ ] 语义化HTML结构

## CMS集中管理系统

### 4.1 CMS系统概述

CMS（内容管理系统）用于集中管理多个静态网站的内容，实现内容的统一维护和更新。

### 4.2 CMS方案对比

#### 4.2.1 WordPress CMS

**技术特点：**
- 基于PHP开发
- 需要将静态网页转换为PHP格式
- 提供直观的后台管理界面

**优点：**
- 界面直观，易于使用
- 插件生态丰富
- 文档完善，社区支持好

**缺点：**
- 需要学习PHP语法
- 静态网页需要转换格式
- 团队需要额外学习成本

**适用场景：**
- 团队已有PHP开发经验
- 需要丰富的插件功能
- 需要快速搭建内容管理后台

#### 4.2.2 Strapi CMS（当前采用方案）

**技术特点：**
- Headless CMS架构
- 基于Node.js开发
- RESTful API接口
- 支持自定义内容类型

**优点：**
- 无需转换静态网页格式
- 灵活的API接口
- 支持多种前端框架
- 可自托管或使用云服务

**缺点：**
- 免费版文字限制250字符
- 云服务需要付费版才能增加文章文字
- 需要一定的技术基础

**适用场景：**
- 需要灵活的API接口
- 多前端框架支持
- 团队有JavaScript基础

#### 4.2.3 方案对比表

| 对比项 | WordPress | Strapi |
|-------|-----------|--------|
| 技术栈 | PHP | Node.js |
| 学习成本 | 中等（需学PHP） | 低（JavaScript） |
| 界面友好度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 灵活性 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 免费版限制 | 较少 | 文字250字符限制 |
| 云服务费用 | 中等 | 较高（付费版） |
| 适用场景 | 传统CMS需求 | 现代Headless CMS |

#### 4.3.2 部署方案

**方案一：Strapi Cloud（推荐用于多人协作）**

**优点：**
- 无需自行维护服务器
- 支持多人协作管理
- 通过链接即可注册登录
- 自动备份和更新

**操作流程：**
1. 注册Strapi Cloud账号
2. 创建新项目
3. 配置内容类型（Content Types）
4. 添加团队成员
5. 通过后台链接邀请成员注册

**方案二：本地部署 + 脚本管理**

**优点：**
- 完全控制数据
- 无云服务费用
- 适合小团队

**操作流程：**
1. 在本地或服务器部署Strapi
2. 配置内容类型
3. 使用脚本同步GitHub和Strapi数据

#### 4.3.3 内容管理方式

**方式一：Strapi云后台管理（推荐）**

**适用场景：** 多人协作、频繁更新内容

**操作流程：**
1. 登录Strapi Cloud后台
2. 在后台直接新增/编辑内容
3. 内容自动同步到GitHub（通过Webhook或脚本）
4. 触发部署流程

**优点：**
- 操作简单直观
- 支持多人同时操作
- 权限管理完善

**方式二：脚本批量管理**

**适用场景：** 批量导入、自动化流程

**操作流程：**

**新增文章：**
1. 在本地准备好文章内容
2. 将文章文件放在指定文件夹
3. 执行文章新增脚本
4. 脚本自动读取文章内容
5. 脚本调用Strapi API创建内容
6. 脚本同步到GitHub

**新增网站：**
1. 完成静态网站开发
2. 将网站文件放在指定位置
3. 执行网站新增脚本
4. 脚本读取网站结构
5. 脚本按照Strapi字段格式创建内容
6. 脚本同步到GitHub

**脚本功能：**
- 自动读取本地文件
- 格式转换（HTML → Strapi格式）
- API调用创建/更新内容
- GitHub同步
- 错误处理和日志记录

#### 4.3.4 Strapi字段配置

**网站内容类型字段：**
- `title` (Text) - 网站标题
- `slug` (Text) - URL标识
- `description` (Text) - 网站描述
- `content` (Rich Text) - 网站内容
- `htmlContent` (Text) - HTML内容
- `images` (Media) - 相关图片
- `category` (Relation) - 分类关联

**文章内容类型字段：**
- `title` (Text) - 文章标题
- `slug` (Text) - URL标识
- `excerpt` (Text) - 文章摘要（250字符限制）
- `content` (Rich Text) - 文章内容
- `htmlContent` (Text) - HTML内容
- `author` (Relation) - 作者关联
- `publishDate` (Date) - 发布日期
- `featuredImage` (Media) - 特色图片

### 4.5 性能优化考虑

#### 4.5.1 内容更新延迟问题

**现状：**
- 静态网页内容存储在GitHub
- 上传新内容需要缓冲时间
- 网站更新内容时加载速度受影响

**解决方案：**

1. **优化静态网页结构**
   - 减少不必要的资源文件
   - 优化图片大小和格式
   - 使用CDN加速

2. **优化生成方式**
   - 预生成静态页面
   - 使用静态站点生成器（SSG）
   - 实现增量更新

3. **使用缓存策略**
   - 浏览器缓存
   - CDN缓存
   - 服务端缓存

#### 4.5.2 工具选择

**GitHub：**
- 极佳的代码仓库选择
- 可串接多个工具
- 目前没有更好的替代品

**Cloudflare：**
- 提供脚本功能
- 可使用原始静态网页代码
- 通过脚本转换到Cloudflare界面
- 支持后台或本机脚本修改

## 代码集中管理

### 5.1 GitHub仓库管理

#### 5.1.1 仓库初始化

**步骤一：创建新仓库**

1. 登录GitHub
2. 点击 "New repository"
3. 输入仓库名称（例如：`multi-site-static-strapi-front`）
4. 勾选 "Initialize with README"
5. 点击 "Create repository"

**步骤二：本地仓库初始化**

在项目根目录执行以下命令：

```bash
# 初始化Git仓库
git init

# 添加所有文件到暂存区
git add .

# 检查远程仓库配置
git remote -v

# 移除旧的远程仓库（如存在）
git remote remove origin

# 添加新的远程仓库
git remote add origin https://github.com/用户名/仓库名.git

# 推送到远程仓库（强制推送，首次使用）
git push -u origin main --force
```

**命令说明：**
- `git init` - 初始化本地Git仓库
- `git add .` - 添加所有文件到暂存区
- `git remote add origin` - 添加远程仓库地址
- `git push -u origin main --force` - 推送到远程仓库（首次使用force）

- `feat` - 新功能

- `docs` - 文档更新
- `style` - 代码格式调整
- `refactor` - 代码重构
- `test` - 测试相关
- `chore` - 构建/工具相关

**示例：**
```
feat: 新增site5网站

- 添加首页、关于、联系页面
- 集成Strapi CMS
- 完成响应式设计
```

#### 5.3.2 分支管理

**主分支：** `main`
- 用于生产环境
- 保持稳定可部署状态

**开发分支：** `develop`
- 用于日常开发
- 功能完成后合并到main

**功能分支：** `feature/功能名称`
- 用于开发新功能
- 完成后合并到develop

### 6.2 Vercel部署流程

#### 6.2.1 首次部署

**步骤一：连接GitHub**

1. 登录Vercel账号
2. 点击 "Add New Project"
3. 点击GitHub图标连接GitHub账号
4. 授权Vercel访问GitHub仓库

**步骤二：选择项目**

1. 在项目列表中找到GitHub仓库（例如：`multi-site-static-strapi-front`）
2. 点击 "Import"

**步骤三：配置项目**

**Framework Preset：** 选择 "Other"

**Root Directory：** 保持默认 `./`

**Build Command：** 留空（静态网站无需构建）

**Output Directory：** 留空（使用根目录）

**步骤四：部署**

1. 检查配置无误
2. 点击 "Deploy"
3. 等待部署完成

**步骤五：获取访问地址**

部署完成后，Vercel会提供访问地址：
- 默认地址：`https://项目名-随机字符.vercel.app`
- 自定义域名：可在设置中配置

#### 6.2.2 网站访问路径

**多站点结构：**
```
https://你的域名.vercel.app/site1/index.html
https://你的域名.vercel.app/site2/index.html
https://你的域名.vercel.app/site3/index.html
```

**示例：**
```
https://multi-site-static-strapi-front.vercel.app/site1/index.html
https://multi-site-static-strapi-front.vercel.app/site2/index.html
```

#### 6.2.3 自动部署

**触发条件：**
- 推送到GitHub main分支
- 手动触发部署

**部署流程：**
```
GitHub推送
    ↓
Vercel检测到更新
    ↓
自动开始部署
    ↓
部署完成
    ↓
更新访问地址
```
