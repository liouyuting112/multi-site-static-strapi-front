## Word文件格式建議

為確保文件清晰易讀，建議在Word中設定以下格式：

### 文字與背景對比度設定

**標題格式：**
- 一級標題：深藍色 (#1a237e) 或黑色，背景白色
- 二級標題：深藍色 (#283593) 或黑色，背景白色
- 三級標題：深灰色 (#424242) 或黑色，背景白色

**正文格式：**
- 文字顏色：黑色 (#000000)
- 背景顏色：白色 (#ffffff)
- 字體大小：建議11-12pt

**表格格式：**
- 標題列：深色背景 (#1976d2) + 白色文字 (#ffffff)
- 內容列：白色背景 + 黑色文字
- 交替行：淺灰色背景 (#f5f5f5) + 黑色文字

**程式碼區塊：**
- 背景顏色：深灰色 (#263238) 或淺灰色 (#f5f5f5)
- 文字顏色：深色背景用淺色文字 (#ffffff)，淺色背景用深色文字 (#000000)

**重點標記：**
- 使用深色粗體文字，避免使用淺色背景
- 如需背景色，使用深色背景配白色文字

**對比度標準：**
- 文字與背景對比度應至少達到 4.5:1（WCAG AA標準）
- 建議使用 7:1 以上對比度以確保最佳可讀性

## 專案概述

### 1.1 專案背景

本專案旨在建立一套完整的靜態網站生成、管理與部署系統，透過AI輔助生成、CMS集中管理、Git版本控制以及雲端部署，實現高效的多站點管理流程。

### 1.2 系統架構

```
┌─────────────────┐
│  AI生成工具     │
│  (Cursor)       │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  靜態網頁檔案   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐      ┌──────────────┐
│  GitHub倉庫     │◄─────┤  CMS後台     │
│  (版本控制)     │      │  (Strapi)    │
└────────┬────────┘      └──────────────┘
         │
         ▼
┌─────────────────┐
│  雲端託管平台   │
│ (Vercel/Netlify)│
└─────────────────┘
```

### 1.3 核心目標

- **高效生成**：透過AI輔助快速生成符合規範的靜態網頁
- **集中管理**：透過CMS系統統一管理多站點內容
- **版本控制**：使用GitHub進行程式碼版本管理
- **快速部署**：自動化部署流程，提升發布效率

## 靜態網頁生成方式

本系統支援三種靜態網頁生成方式，可根據專案需求選擇合適的方法。

### 2.1 方式一：AI直接生成

#### 2.1.1 流程說明

**適用場景：** 全新網站、無特定模板要求

**操作流程：**
1. 在Cursor中直接向AI提供需求描述
2. 明確說明網站主題、風格、功能需求
3. AI根據提示詞生成完整的靜態網頁程式碼
4. 檢查生成結果是否符合要求

**優點：**
- 生成速度快
- 靈活性高
- 適合快速原型開發

**注意事項：**
- 需要提供清晰、詳細的需求描述
- 生成後需進行品質檢查
- 可能需要多次迭代優化

#### 2.1.2 提示詞要求

- 明確網站主題和定位
- 說明所需頁面結構（首頁、關於、文章列表等）
- 指定設計風格（現代、簡約、商務等）
- 列出必需功能模組

---

### 2.2 方式二：基於HTML模板生成

#### 2.2.1 流程說明

**適用場景：** 需要保持特定架構結構、有現成模板

**操作流程：**
1. 下載或準備HTML靜態網頁模板
2. 將模板檔案提供給Cursor
3. 向AI說明網站主題
4. 提供文章內容要求：
   - 文章必須包含的重點內容
   - 文章格式要求（標題層級、段落結構等）
   - 內容長度要求
5. AI根據模板結構生成符合架構的靜態網頁
6. 驗證生成結果與模板結構的一致性

**優點：**
- 保持統一的網站架構
- 結構規範，易於維護
- 適合批量生成相似結構的網站

**注意事項：**
- 模板品質直接影響生成效果
- 需要確保模板結構清晰
- 生成後需驗證結構一致性

#### 2.2.2 模板要求

- 模板應包含完整的HTML結構
- 建議包含CSS樣式檔案
- 模板應具備良好的程式碼註釋
- 模板應遵循響應式設計原則

---

### 2.3 方式三：基於Figma設計稿生成

#### 2.3.1 流程說明

**適用場景：** 有完整設計稿、需要精確還原設計

**操作流程：**
1. 在Figma中完成網站設計
2. 將Figma設計稿匯出為圖片（PNG/JPG）
3. 將設計圖片提供給Cursor
4. 向AI說明需要根據設計稿生成靜態網頁
5. 提出需要新增的功能專案（如：每日文章更新模組）
6. AI根據設計稿樣式生成對應的靜態網頁
7. 對比生成結果與設計稿的一致性

**優點：**
- 設計還原度高
- 視覺呈現精確
- 適合有明確設計要求的專案

**注意事項：**
- 設計稿應清晰完整
- 需要提供設計規範說明
- 生成後需進行視覺對比檢查

#### 2.3.2 設計稿要求

- 設計稿應包含所有頁面狀態
- 建議提供設計規範（顏色、字體、間距等）
- 標註互動說明
- 提供響應式斷點設計

## 靜態網頁品質檢查

生成靜態網頁後，需要進行全面的品質檢查，確保網站符合發布標準。

### 3.1 檢查專案清單

#### 3.1.1 結構檢查

**HTML結構驗證：**
- [ ] DOCTYPE宣告正確
- [ ] HTML標籤閉合完整
- [ ] 語義化標籤使用正確（header, nav, main, article, footer等）
- [ ] 頁面層級結構清晰
- [ ] 連結路徑正確（相對路徑/絕對路徑）
- [ ] 圖片路徑正確
- [ ] 腳本檔案引用正確

**檔案結構驗證：**
- [ ] 檔案命名規範統一
- [ ] 目錄結構清晰合理
- [ ] CSS檔案組織規範
- [ ] JavaScript檔案組織規範
- [ ] 資源檔案分類存放

#### 3.1.2 響應式設計檢查

**斷點測試：**
- [ ] 桌面端顯示正常（1920px, 1440px, 1280px）
- [ ] 平板端顯示正常（768px, 1024px）
- [ ] 行動端顯示正常（375px, 414px）
- [ ] 橫屏/豎屏切換正常

**佈局檢查：**
- [ ] 內容不溢出容器
- [ ] 文字大小適配不同螢幕
- [ ] 圖片自適應顯示
- [ ] 導航選單在小螢幕正常顯示
- [ ] 表格在小螢幕可橫向滾動

**互動檢查：**
- [ ] 按鈕點擊區域足夠大（行動端）
- [ ] 表單輸入框大小合適
- [ ] 觸摸操作流暢

#### 3.1.3 內容檢查

**文章內容：**
- [ ] 文章標題清晰明確
- [ ] 文章內容完整，包含所有必需重點
- [ ] 段落結構合理
- [ ] 標題層級正確（H1-H6）
- [ ] 列表格式正確
- [ ] 引用格式正確
- [ ] 圖片alt屬性完整

**頁面內容：**
- [ ] 首頁內容完整
- [ ] 關於頁面資訊準確
- [ ] 聯絡頁面表單功能正常
- [ ] 文章列表顯示正確
- [ ] 文章詳情頁內容完整

#### 3.1.4 效能檢查

**載入速度：**
- [ ] 首屏載入時間 < 3秒
- [ ] 圖片優化（WebP格式、適當壓縮）
- [ ] CSS/JS檔案壓縮
- [ ] 無冗餘程式碼

**資源優化：**
- [ ] 圖片懶載入
- [ ] 字體檔案優化
- [ ] 第三方資源CDN載入

#### 3.1.5 相容性檢查

**瀏覽器相容：**
- [ ] Chrome最新版
- [ ] Firefox最新版
- [ ] Safari最新版
- [ ] Edge最新版
- [ ] 行動端瀏覽器（iOS Safari, Chrome Mobile）

**功能相容：**
- [ ] JavaScript功能正常
- [ ] CSS動畫效果正常
- [ ] 表單提交功能正常

#### 3.1.6 SEO檢查

**基礎SEO：**
- [ ] 頁面title標籤完整
- [ ] meta description設定
- [ ] meta keywords設定（如需要）
- [ ] 語義化HTML結構
- [ ] 圖片alt屬性完整
- [ ] 連結title屬性（如需要）

**結構化資料：**
- [ ] sitemap.xml檔案存在
- [ ] robots.txt檔案設定正確

## CMS集中管理系統

### 4.1 CMS系統概述

CMS（內容管理系統）用於集中管理多個靜態網站的內容，實現內容的統一維護和更新。

### 4.2 CMS方案對比

#### 4.2.1 WordPress CMS

**技術特點：**
- 基於PHP開發
- 需要將靜態網頁轉換為PHP格式
- 提供直觀的後台管理介面

**優點：**
- 介面直觀，易於使用
- 外掛生態豐富
- 文件完善，社群支援好

**缺點：**
- 需要學習PHP語法
- 靜態網頁需要轉換格式
- 團隊需要額外學習成本

**適用場景：**
- 團隊已有PHP開發經驗
- 需要豐富的外掛功能
- 需要快速搭建內容管理後台

#### 4.2.2 Strapi CMS（目前採用方案）

**技術特點：**
- Headless CMS架構
- 基於Node.js開發
- RESTful API介面
- 支援自訂內容類型

**優點：**
- 無需轉換靜態網頁格式
- 靈活的API介面
- 支援多種前端框架
- 可自託管或使用雲服務

**缺點：**
- 免費版文字限制250字元
- 雲服務需要付費版才能增加文章文字
- 需要一定的技術基礎

**適用場景：**
- 需要靈活的API介面
- 多前端框架支援
- 團隊有JavaScript基礎

#### 4.2.3 方案對比表

| 對比項 | WordPress | Strapi |
|-------|-----------|--------|
| 技術棧 | PHP | Node.js |
| 學習成本 | 中等（需學PHP） | 低（JavaScript） |
| 介面友好度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 靈活性 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 免費版限制 | 較少 | 文字250字元限制 |
| 雲服務費用 | 中等 | 較高（付費版） |
| 適用場景 | 傳統CMS需求 | 現代Headless CMS |

#### 4.3.2 部署方案

**方案一：Strapi Cloud（推薦用於多人協作）**

**優點：**
- 無需自行維護伺服器
- 支援多人協作管理
- 透過連結即可註冊登入
- 自動備份和更新

**操作流程：**
1. 註冊Strapi Cloud帳號
2. 建立新專案
3. 設定內容類型（Content Types）
4. 新增團隊成員
5. 透過後台連結邀請成員註冊

**方案二：本地部署 + 腳本管理**

**優點：**
- 完全控制資料
- 無雲服務費用
- 適合小團隊

**操作流程：**
1. 在本地或伺服器部署Strapi
2. 設定內容類型
3. 使用腳本同步GitHub和Strapi資料

#### 4.3.3 內容管理方式

**方式一：Strapi雲後台管理（推薦）**

**適用場景：** 多人協作、頻繁更新內容

**操作流程：**
1. 登入Strapi Cloud後台
2. 在後台直接新增/編輯內容
3. 內容自動同步到GitHub（透過Webhook或腳本）
4. 觸發部署流程

**優點：**
- 操作簡單直觀
- 支援多人同時操作
- 權限管理完善

**方式二：腳本批量管理**

**適用場景：** 批量匯入、自動化流程

**操作流程：**

**新增文章：**
1. 在本地準備好文章內容
2. 將文章檔案放在指定資料夾
3. 執行文章新增腳本
4. 腳本自動讀取文章內容
5. 腳本呼叫Strapi API建立內容
6. 腳本同步到GitHub

**新增網站：**
1. 完成靜態網站開發
2. 將網站檔案放在指定位置
3. 執行網站新增腳本
4. 腳本讀取網站結構
5. 腳本按照Strapi欄位格式建立內容
6. 腳本同步到GitHub

**腳本功能：**
- 自動讀取本地檔案
- 格式轉換（HTML → Strapi格式）
- API呼叫建立/更新內容
- GitHub同步
- 錯誤處理和日誌記錄

#### 4.3.4 Strapi欄位設定

**網站內容類型欄位：**
- `title` (Text) - 網站標題
- `slug` (Text) - URL識別碼
- `description` (Text) - 網站描述
- `content` (Rich Text) - 網站內容
- `htmlContent` (Text) - HTML內容
- `images` (Media) - 相關圖片
- `category` (Relation) - 分類關聯

**文章內容類型欄位：**
- `title` (Text) - 文章標題
- `slug` (Text) - URL識別碼
- `excerpt` (Text) - 文章摘要（250字元限制）
- `content` (Rich Text) - 文章內容
- `htmlContent` (Text) - HTML內容
- `author` (Relation) - 作者關聯
- `publishDate` (Date) - 發布日期
- `featuredImage` (Media) - 特色圖片

### 4.5 效能優化考量

#### 4.5.1 內容更新延遲問題

**現狀：**
- 靜態網頁內容儲存在GitHub
- 上傳新內容需要緩衝時間
- 網站更新內容時載入速度受影響

**解決方案：**

1. **優化靜態網頁結構**
   - 減少不必要的資源檔案
   - 優化圖片大小和格式
   - 使用CDN加速

2. **優化生成方式**
   - 預生成靜態頁面
   - 使用靜態網站生成器（SSG）
   - 實現增量更新

3. **使用快取策略**
   - 瀏覽器快取
   - CDN快取
   - 伺服器端快取

#### 4.5.2 工具選擇

**GitHub：**
- 極佳的程式碼倉庫選擇
- 可串接多個工具
- 目前沒有更好的替代品

**Cloudflare：**
- 提供腳本功能
- 可使用原始靜態網頁程式碼
- 透過腳本轉換到Cloudflare介面
- 支援後台或本機腳本修改

## 程式碼集中管理

### 5.1 GitHub倉庫管理

#### 5.1.1 倉庫初始化

**步驟一：建立新倉庫**

1. 登入GitHub
2. 點擊 "New repository"
3. 輸入倉庫名稱（例如：`multi-site-static-strapi-front`）
4. 勾選 "Initialize with README"
5. 點擊 "Create repository"

**步驟二：本地倉庫初始化**

在專案根目錄執行以下命令：

```bash
# 初始化Git倉庫
git init

# 新增所有檔案到暫存區
git add .

# 檢查遠端倉庫設定
git remote -v

# 移除舊的遠端倉庫（如存在）
git remote remove origin

# 新增新的遠端倉庫
git remote add origin https://github.com/使用者名稱/倉庫名.git

# 推送到遠端倉庫（強制推送，首次使用）
git push -u origin main --force
```

**命令說明：**
- `git init` - 初始化本地Git倉庫
- `git add .` - 新增所有檔案到暫存區
- `git remote add origin` - 新增遠端倉庫位址
- `git push -u origin main --force` - 推送到遠端倉庫（首次使用force）

### 6.2 Vercel部署流程

#### 6.2.1 首次部署

**步驟一：連接GitHub**

1. 登入Vercel帳號
2. 點擊 "Add New Project"
3. 點擊GitHub圖示連接GitHub帳號
4. 授權Vercel存取GitHub倉庫

**步驟二：選擇專案**

1. 在專案清單中找到GitHub倉庫（例如：`multi-site-static-strapi-front`）
2. 點擊 "Import"

**步驟三：設定專案**

**Framework Preset：** 選擇 "Other"

**Root Directory：** 保持預設 `./`

**Build Command：** 留空（靜態網站無需建置）

**Output Directory：** 留空（使用根目錄）

**步驟四：部署**

1. 檢查設定無誤
2. 點擊 "Deploy"
3. 等待部署完成

**步驟五：取得存取位址**

部署完成後，Vercel會提供存取位址：
- 預設位址：`https://專案名-隨機字元.vercel.app`
- 自訂網域：可在設定中設定

#### 6.2.2 網站存取路徑

**多站點結構：**
```
https://你的網域.vercel.app/site1/index.html
https://你的網域.vercel.app/site2/index.html
https://你的網域.vercel.app/site3/index.html
```

**範例：**
```
https://multi-site-static-strapi-front.vercel.app/site1/index.html
https://multi-site-static-strapi-front.vercel.app/site2/index.html
```

GitHub推送

部署完成
