## 同一網站生成不同 CSS 樣式

### 功能說明

此功能可為同一個網站生成多個不同視覺風格的版本，適用於以下情境：

- **A/B 測試**：測試不同設計風格對使用者體驗的影響
- **主題變體**：為同一網站建立多個主題版本（例如：深色模式、淺色模式）
- **客戶展示**：向客戶展示多種設計選項

### 適用時機

✅ **建議使用**：
- 網站基本結構已確定
- 需要測試不同視覺風格
- 客戶要求多個設計選項

❌ **不建議使用**：
- 網站尚未完成基本開發
- 僅需要單一版本
- 時間與資源有限

### 操作步驟

#### 步驟一：準備原始網站

1. 確認原始網站已完成品質檢查
2. 確認網站結構符合標準
3. 備份原始網站檔案（建議）

#### 步驟二：使用 Cursor Composer 生成

1. 開啟 Cursor 編輯器
2. 使用 Cursor Composer 功能（快捷鍵：`Ctrl+I`）
3. 參考提詞範本：`下載/一個網站生成多個樣式.txt`
4. 提供以下資訊：

**提詞範例**：

```
請為以下網站生成新的 CSS 樣式版本：

原始網站位置：[網站資料夾完整路徑]
目標生成位置：[新資料夾完整路徑]

樣式修改需求：
1. 主色調改為：[顏色名稱或色碼]
2. 字體改為：[字體名稱]
3. 佈局風格改為：[描述，例如：更緊湊、更寬鬆]
4. 其他特殊需求：[如有]

請保持網站結構與功能不變，僅修改視覺樣式。
```

#### 步驟三：檢查生成結果

1. 開啟生成的網站資料夾
2. 檢查所有頁面是否正常顯示
3. 進行基本的品質檢查（參考主規格書「靜態網頁品質檢查」章節）
4. 確認樣式變更符合需求

#### 步驟四：上傳與部署

1. 將新樣式版本上傳到 GitHub（使用不同的資料夾名稱）
2. 如需要，可設定為不同的子網域或路徑
3. 部署到 Vercel 進行測試

### 注意事項

⚠️ **重要提醒**：

1. **檔案管理**：建議使用不同的資料夾名稱區分不同版本
   - 範例：`星宿探索家sce010-深色版`、`星宿探索家sce010-淺色版`

2. **內容同步**：不同樣式版本的內容需要分別管理
   - 建議：使用相同的 CMS 後台，但建立不同的站點記錄

3. **維護成本**：多個版本會增加維護工作量
   - 每次內容更新需要同步到所有版本
   - 建議僅在必要時使用此功能

4. **品質檢查**：每個版本都需要獨立進行品質檢查
   - 不能因為是「樣式變體」就省略檢查步驟

## 圖片自動偵測與補齊

### 功能說明

此功能可自動偵測網站中缺少的圖片，並從 Pexels 下載或使用 AI 生成補齊。

### 適用時機

✅ **建議使用**：
- 網站中有多處缺少圖片
- 需要批量補齊圖片
- 圖片來源統一（例如：全部使用 Pexels）

❌ **不建議使用**：
- 僅缺少 1-2 張圖片（手動處理更快）
- 需要特定版權的圖片
- 圖片需求非常特殊（AI 生成可能不符合）

### 操作步驟

#### 步驟一：準備圖片描述檔

1. 檢查網站中缺少的圖片位置
2. 為每個缺少的圖片撰寫描述
3. 建立圖片描述檔（文字檔案，每行一個描述）

**描述檔範例**（`缺少圖片清單.txt`）：

```
首頁橫幅圖片：星座運勢主題，深藍色背景，現代風格
關於我們頁面圖片：團隊合作，專業氛圍
文章配圖1：今日運勢，溫暖色調
文章配圖2：星座分析，神秘風格
```

#### 步驟二：使用 Cursor Composer 偵測

1. 開啟 Cursor 編輯器
2. 使用 Cursor Composer 功能
3. 提供以下提詞：

**提詞範例**：

```
請檢查以下網站中缺少的圖片：

Github圖片位置替換為：https://raw.githubusercontent.com/test100web/100-website/main/shared-assets
本機靜態網頁程式碼位置替換為：C:\Users\yyutingliu\Downloads\AI生成網站測試\cursor\一個主題多個站(落地)\星宿探索家sce010

並提供缺少圖片的描述檔：
[貼上圖片描述檔內容]
```

#### 步驟三：執行圖片下載

1. 前往專案中的「圖片偵測測試」資料夾
2. 閱讀「注意事項(抓取前閱讀).txt」
3. 修改 Pexels API Key（如需要）
4. 執行「一鍵檢測並下載.bat」
5. 選擇圖片清單檔案（由 Cursor 生成或手動建立）

#### 步驟四：檢查與確認

1. 檢查下載的圖片是否符合需求
2. 確認圖片路徑正確
3. 更新網站中的圖片路徑（如需要）
4. 重新檢查網站顯示是否正常

### 圖片來源選擇

#### 方式一：Pexels（推薦）

**優點**：
- 免費使用
- 高品質圖片
- 無版權問題

**操作**：
1. 使用「圖片偵測測試/AI抓取無版權圖(Pexels)」工具
2. 執行「圖片下載.bat」

#### 方式二：AI 生成

**優點**：
- 可生成特定需求的圖片
- 完全客製化

**缺點**：
- 需要 AI 工具（例如：Google AI Studio Nano Banana）
- 生成時間較長

**操作**：
1. 將圖片描述檔上傳到 Google AI Studio
2. 等待生成並下載
3. 手動放置到對應資料夾

### 注意事項

⚠️ **重要提醒**：

1. **版權問題**：確保使用的圖片來源合法
   - Pexels：可免費使用，無需標註來源（建議標註）
   - AI 生成：需確認 AI 工具的授權條款

2. **圖片品質**：下載後檢查圖片品質
   - 解析度是否足夠
   - 檔案大小是否合理
   - 圖片內容是否符合需求

3. **路徑設定**：確認圖片路徑正確
   - 相對路徑：適用於 GitHub 部署
   - 絕對路徑：適用於本機測試

## 批量處理多個網站

### 功能說明

此功能可同時處理多個網站，適用於批量操作情境。

### 適用時機

✅ **建議使用**：
- 需要同時更新多個網站
- 批量上傳新網站到 CMS
- 批量生成文章

❌ **不建議使用**：
- 僅處理 1-2 個網站（單獨處理更快）
- 需要針對不同網站做不同設定

### 批量上傳網站到 CMS

#### 使用批量上傳腳本

**檔案位置**：`批量上傳所有網站.cjs`

**功能**：
- 自動掃描專案中的所有網站資料夾
- 逐一上傳每個網站到 Strapi CMS
- 顯示上傳進度與結果統計

**使用方式**：

```bash
node 批量上傳所有網站.cjs [環境]
```

**參數說明**：
- `環境`：`production`（預設）或 `development`

**範例**：

```bash
# 批量上傳所有網站到正式環境
node 批量上傳所有網站.cjs production

# 批量上傳所有網站到開發環境
node 批量上傳所有網站.cjs development
```

**輸出範例**：

```
🚀 開始批量上傳所有網站到 Strapi CMS

📌 找到 15 個網站資料夾

📄 處理網站 1/15: 星宿探索家sce010
  ✅ 上傳完成：4 個 Page，3 個 Post

📄 處理網站 2/15: 星座解密站cds006
  ✅ 上傳完成：4 個 Page，5 個 Post

...

📊 批量上傳結果統計
總網站數：15
成功：15
失敗：0
總 Page 數：60
總 Post 數：45
```

### 批量生成文章

#### 使用 AI 生成腳本

**檔案位置**：`ai-generate-articles.cjs`

**批量生成設定**：

```bash
# Windows PowerShell
# 設定要生成的站點（多個站點用逗號分隔）
$env:SITES="sce010,site1,cds006,seh001"

# 設定生成數量（每個站點生成幾篇）
$env:COUNT="2"

# 設定文章類別
$env:CATEGORY="daily"

# 設定日期
$env:DATE="2025-01-15"

# 執行生成
node ai-generate-articles.cjs
```

**生成所有站點**：

```bash
# 設定為 all 可生成所有站點
$env:SITES="all"
$env:COUNT="1"
$env:CATEGORY="daily"
$env:DATE="2025-01-15"

node ai-generate-articles.cjs
```

### 批量匯出文章

#### 使用匯出腳本

**檔案位置**：`export-strapi-to-github.cjs`

**批量匯出設定**：

```bash
# 匯出所有站點今天的每日文章
node export-strapi-to-github.cjs all 2025-01-15 daily

# 匯出所有站點所有類別的文章
node export-strapi-to-github.cjs all 2025-01-15 all
```

### 注意事項

⚠️ **重要提醒**：

1. **執行時間**：批量操作可能需要較長時間
   - 建議在非尖峰時段執行
   - 確保網路連線穩定

2. **錯誤處理**：批量操作中如有單一站點失敗
   - 腳本會繼續處理其他站點
   - 最後會顯示失敗的站點清單
   - 需要手動檢查並重新處理失敗的站點

3. **資源消耗**：批量操作會消耗較多資源
   - API 呼叫次數（Strapi、Gemini）
   - 網路頻寬
   - 建議分批處理，避免一次性處理過多站點

4. **備份建議**：執行批量操作前建議備份
   - 備份 GitHub 倉庫
   - 備份 Strapi 資料（如可能）
